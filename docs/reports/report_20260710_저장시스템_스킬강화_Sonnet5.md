## 2. 저장/불러오기 (`SaveSystem`)

- `user://save.json`에 레벨·XP·미배분 스탯/스킬 포인트·1차 스탯·장비 보너스 합계·인벤토리 아이템 Id·스킬 강화 횟수를 JSON으로 저장.
- 저장 트리거: 레벨업, 아이템 픽업, 스탯 분배, 스킬 강화 — 상태가 바뀌는 모든 지점.
- **안전장치**: `OS.HasFeature("standalone")` 기준으로 **실행 파일(.exe)에서만** 자동 저장/불러오기가 켜진다. 에디터·헤드리스 테스트 실행에서는 꺼져 있어, 개발 중 스크린샷 검증이나 회귀 테스트가 사용자의 실제 플레이 진행도(`save.json`)를 절대 덮어쓰지 않는다.
  - 테스트가 저장 경로 자체를 검증해야 할 때는 `Player.ForceEnableSaveForTests` + `SaveSystem.FileName`을 전용 파일명(`test_save.json`)으로 바꿔 완전히 격리한다(`tests/SaveLoadTest`).
  - 실행 후 유저 데이터 폴더를 직접 확인해 실제로 `save.json`이 생기지 않음(패시브 부팅 시)과 테스트 파일이 잔류하지 않음을 검증했다.

## 3. ★ 저장 구현 중 발견한 실전 버그 ★

**증상**: `SaveLoadTest`에서 스킬 강화 배율이 저장된 값의 정확히 2배(0.30, 기대값 0.15)로 로드됨.

**원인**: `Player.tscn`이 `StrikeSkill` 등을 `.tres` 파일의 `ExtResource`로 참조하는데, Godot은 같은 경로의 리소스를 **캐시해 여러 씬 인스턴스가 공유**한다. `UpgradeSkill()`이 이 공유 객체의 `DamageMultiplier`를 직접 변형했기 때문에:
1. 플레이어 A가 강화 → 공유된 `.tres` 객체 자체가 영구히 바뀜.
2. 플레이어 A가 죽고 `R`로 씬을 리로드(`ReloadCurrentScene`)하면, 새 플레이어 B가 **이미 바뀐 공유 객체**를 물려받음.
3. B의 `_Ready()`가 저장된 강화 횟수를 그 위에 **또** 더함 → 이중 적용.

이건 세이브 기능이 없던 이전 버전에서도 잠재해 있던 문제로(R 재시작마다 스킬이 몰래 계속 강해짐), 저장 기능을 만들며 작성한 테스트가 우연히 잡아낸 것이다. **수정**: `_Ready()`에서 `StrikeSkill/CrushSkill/SpinSkill`을 인스턴스별로 `Duplicate()`해 각 Player가 독립된 SkillData 객체를 갖도록 격리.

## 4. 스킬 포인트 실사용

- 이전까지 "스킬 포인트"는 HUD에 숫자만 쌓이고 아무 효과가 없었다.
- 키 `8`/`9`/`0` = 강타/분쇄/회전격돌 각각 강화. 포인트 1개당 `DamageMultiplier +0.15`(`BalanceConstants.SkillUpgradeDamageBonus`).
- 스탯 분배(`alloc_str` 등)도 이번에 저장 트리거를 추가— 이전엔 스탯만 분배하고 아무 것도 안 하면 다음 저장 시점까지 유실될 수 있었다(별도 발견·수정).
- HUD 안내 문구에 강화 키 안내 추가.
